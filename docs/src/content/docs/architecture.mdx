## Three-tier Command Execution

```
command arrives at Workspace DO
        │
        ▼
  Is it a built-in? (cat, ls, grep -r, echo, pwd, ...)
        │
        ├── YES → Tier 1: Execute in DO directly
        │         $0 cost, <1ms latency
        │         Reads/writes via FsEngine
        │
        └── NO
            │
            ▼
      Is it JS/TS? (node, ./script.js, npx)
            │
            ├── YES → Tier 2: JsRunner in DO
            │         $0 cost, ~ms latency
            │         V8 isolate with module system
            │         require(), import/export, TypeScript
            │         Node.js builtins: fs, path, crypto, http,
            │         worker_threads, stream, net
            │
            └── NO → Tier 3: Route to Container
                      ~$0.02/hr, real Linux filesystem
                      ctx.container.getTcpPort(8080).fetch()
                      Container runs command → returns result → sleeps
```

### JsRunner (Tier 2)

The JsRunner executes JavaScript and TypeScript directly in the DO's V8 isolate:

- **Module system**: CommonJS `require()` and ESM `import/export` (auto-converted to CJS)
- **Type stripping**: TypeScript annotations stripped at load time (no tsc needed)
- **Circular deps**: handled correctly with module cache
- **Node.js builtins**: `fs` (backed by FsEngine), `path`, `crypto` (createHash), `http` (backed by fetch), `worker_threads` (isolated JsRunner instances), `stream`, `net`
- **Real http.request()**: ClientRequest class collects body chunks, calls fetch(), wraps Response as IncomingMessage with statusCode/headers/data events
- **Real worker_threads**: Worker creates an isolated JsRunner with its own module cache, bidirectional postMessage with structured clone

### Shell Parsing Pipeline

Before dispatch, the command string is parsed for shell operators:

```
"cat file | grep ERROR && echo found || echo missing"
        │
        ▼
  splitPipeline() → ["cat file", "grep ERROR && echo found || echo missing"]
        │
        ▼
  For pipe segments without pipes:
    splitChain() → [{cmd: "grep ERROR", op: ""}, {cmd: "echo found", op: "&&"}, ...]
        │
        ▼
  Each segment dispatched to execSingle()
    → Tier 1 (builtin) or Tier 2 (JsRunner) or Tier 3 (container)
```

Pipes feed stdout→stdin between stages. Chains evaluate `&&` (stop on fail), `||` (run on fail), and `;` (always run).

## Three-tier File Cache

```
┌──────────────────────────────────────────────────────┐
│ Hot:  zerobuf WASM Memory    (sub-μs, zero-copy)     │
│       WebAssembly.Memory arena, Uint8Array views     │
│       32MB budget, survives across requests           │
├──────────────────────────────────────────────────────┤
│ Warm: SQLite file_cache      (<1ms, persistent)      │
│       Files < 64KB, max 500 entries                  │
│       Survives DO hibernation/eviction               │
├──────────────────────────────────────────────────────┤
│ Cold: R2                     (10-50ms, durable)      │
│       Unlimited storage, $0.015/GB/month             │
│       Global distribution                            │
└──────────────────────────────────────────────────────┘
```

Reads check hot → warm → cold, promoting upward on miss. Writes invalidate + re-populate the hot cache. The hot tier uses [zerobuf](https://www.npmjs.com/package/zerobuf) to store file content in WebAssembly linear memory — reads return `Uint8Array` views directly into WASM memory with no copies.

### R2 for File Content

Files are stored as R2 objects keyed by `{workspace_id}/{path}`. R2 provides:
- Unlimited storage ($0.015/GB/month)
- Streaming reads/writes (128MB Worker memory limit safe)
- Global distribution

### DO SQLite for Metadata

R2 is a flat key-value store — it has no directory concept. DO SQLite provides:
- **Directory index** (path → size, mode, mtime, is_dir)
- **Warm file cache** (files &lt; 64KB cached in SQLite for &lt;1ms reads vs R2's 10-50ms)
- **Process table** (last 1000 commands, output truncated to 4KB)
- **Terminal buffer** (last 5000 rows, persists output across DO hibernation)

### Why This Split?

| Concern | zerobuf (WASM) | SQLite | R2 |
|---------|---------------|--------|----|
| Speed | sub-μs | &lt;1ms | 10-50ms |
| Persistence | In-memory only | Survives hibernation | Durable |
| Capacity | 32MB | ~10GB | Unlimited |
| Copy cost | Zero-copy views | Copy to JS | Copy to JS |
| Best for | Hot reads, grep | Warm reads, metadata | Cold storage |

## Container Integration

Non-builtin, non-JS commands run in a Cloudflare Container attached to the DO via `ctx.container`:

```
┌─────────────────────────────────────┐
│ Workspace Durable Object            │
│                                     │
│  ┌──────────┐  ┌─────────────────┐  │     ┌──────────────────────┐
│  │ FsEngine │  │ ProcessManager  │──┼────▶│ Container            │
│  │(R2+SQL+  │  │ (builtins +     │  │     │                      │
│  │ WASM)    │  │  JsRunner)      │  │     │  agent.mjs :8080     │
│  └──────────┘  └─────────────────┘  │     │  ├── /exec            │
│                                     │     │  ├── /healthz         │
│  ctx.container.getTcpPort(8080)     │     │  └── /sync/snapshot   │
│                                     │     │                      │
│  alarm() → health check            │     │  /workspace/ (symlinks)│
│          → index reconciliation     │     │  /mnt/workspace (FUSE)│
│                                     │     │  /local/ (fast disk)  │
│  WebSocket stdio                    │     │                      │
└─────────────────────────────────────┘     └──────────────────────┘
```

### Container Lifecycle

1. **First exec** → `container.start()` with env vars, `container.monitor()` to wait
2. **Running** → Commands via `getTcpPort(8080).fetch("/exec")`
3. **Idle 10 min** → `setInactivityTimeout()` puts container to sleep
4. **SIGTERM** → agent snapshots `node_modules`/`dist` as tar.zst to R2
5. **Next exec** → Container boots, restores snapshots, ready in seconds
6. **Health check** → DO alarm every 30s: ping + R2↔SQLite index reconciliation

### File Sync Model

| Files | Location | Speed | Persistence |
|-------|----------|-------|-------------|
| Source files | R2 FUSE mount → `/mnt/workspace` | Medium | Permanent |
| Build artifacts | Container local → `/local/` | Fast | Snapshot to R2 on sleep |
| SQLite index | DO SQLite | &lt;1ms | Permanent |
| Hot cache | zerobuf WASM Memory | sub-μs | In-memory only |

## Workspace Isolation

Each workspace is a separate Durable Object instance:
- Own SQLite database (filesystem index, process table)
- Own R2 prefix (`{workspace_id}/`)
- Own WASM Memory content cache (zerobuf arena)
- Own Container instance
- Own WebSocket connections
- Single-threaded (no locks needed)
- Hibernates when idle (zero cost)

## Input Validation

All inputs validated at system boundaries:
- **Paths**: max 4096 chars, max 256 depth, no null bytes, no traversal above root
- **Commands**: max 8192 chars, no null bytes
- **Workspace IDs**: alphanumeric + `.`, `-`, `_`, max 128 chars
- **Payloads**: max 100MB (Worker), max 10MB (container agent)
- **Regex**: invalid/unsafe grep patterns fall back to literal string matching

## Composition with Other *mode Packages

```
termweb workspace
├── gitmode   — git storage (R2 + DO SQLite)
├── nodemode  — Node.js runtime (R2 fs + DO JsRunner + Container)
├── pymode    — Python runtime (CPython WASM + DO fan-out)
├── querymode — SQL queries (columnar engine + R2 spill)
└── workspace — orchestrator (auth + routing + lifecycle)
```

Each package is independent and composable via service bindings or direct import.
