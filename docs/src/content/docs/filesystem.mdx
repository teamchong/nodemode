## Overview

nodemode's `FsEngine` maps Node.js `fs` operations to a three-tier storage stack: zerobuf WASM Memory (hot) → DO SQLite (warm) → R2 (cold).

## Supported Operations

| Node.js fs | nodemode | Backing |
|-----------|----------|---------|
| `readFile` | `fs.readFile(path)` | WASM cache → SQLite cache → R2 |
| `readFile` (stream) | `fs.readFileStream(path)` | R2 streaming body |
| `writeFile` | `fs.writeFile(path, data)` | R2 put + SQLite index + caches |
| `appendFile` | `fs.appendFile(path, data)` | Read + concat + write |
| `unlink` | `fs.unlink(path)` | R2 delete + SQLite delete + cache invalidate |
| `stat` | `fs.stat(path)` | SQLite lookup |
| `exists` | `fs.exists(path)` | SQLite lookup |
| `readdir` | `fs.readdir(path)` | SQLite prefix query |
| `mkdir` | `fs.mkdir(path, recursive?)` | SQLite directory marker |
| `rmdir` | `fs.rmdir(path, recursive?)` | R2 batch delete + SQLite + cache clear |
| `rename` | `fs.rename(old, new)` | R2 copy+delete + SQLite + cache clear |
| `chmod` | `fs.chmod(path, mode)` | SQLite update |
| `touch` | `fs.touch(path)` | SQLite mtime update |

## Three-tier Caching

```
┌──────────────────────────────────────────────────────┐
│ Hot:  zerobuf WASM Memory    (sub-μs, zero-copy)     │
│       Arena allocator in WebAssembly.Memory          │
│       32MB budget, Uint8Array views into WASM        │
│       Survives across requests, lost on DO eviction  │
├──────────────────────────────────────────────────────┤
│ Warm: SQLite file_cache      (<1ms, persistent)      │
│       Files < 64KB, max 500 entries                  │
│       Survives DO hibernation/eviction               │
├──────────────────────────────────────────────────────┤
│ Cold: R2                     (10-50ms, durable)      │
│       Unlimited storage, $0.015/GB/month             │
│       Global distribution                            │
└──────────────────────────────────────────────────────┘
```

### How it works

**Reads** check hot → warm → cold, promoting upward on miss:
1. Check zerobuf WASM Memory — return zero-copy `Uint8Array` view (sub-μs)
2. Check SQLite `file_cache` — copy to JS, promote to hot cache (&lt;1ms)
3. Fetch from R2 — populate both hot and warm caches (10-50ms)

**Writes** invalidate + re-populate:
1. Write to R2 (durable)
2. Invalidate old hot cache entry, store new content in WASM arena
3. Update SQLite warm cache (if file &lt; 64KB)

**Deletes** invalidate all tiers:
1. Delete from R2
2. Remove from SQLite index and cache
3. Invalidate hot cache entry

### zerobuf integration

The hot tier uses [zerobuf](https://www.npmjs.com/package/zerobuf)'s Arena allocator over `WebAssembly.Memory`:
- `Arena.allocBytes(data)` stores file content directly in WASM linear memory
- Reads return `Uint8Array` views pointing into WASM memory — **no copies**
- Arena is append-only with save/restore checkpoints
- Automatic memory growth (doubling strategy, up to 32MB)

This eliminates the `ArrayBuffer → Uint8Array → TextDecoder` copy chain for cached files. For `grep -r` scanning many files, content is already in memory from the first read.

## REST API

### Write a file

```
POST /workspace/{id}/fs/write
Content-Type: application/json

{"path": "src/index.ts", "content": "console.log('hello');", "mode": 420}
```

### Read a file

```
POST /workspace/{id}/fs/read
Content-Type: application/json

{"path": "src/index.ts"}
```

Response: `{"content": "console.log('hello');"}`

### Read a file (streaming)

For files larger than Worker memory limits, use streaming mode:

```
POST /workspace/{id}/fs/read
Content-Type: application/json

{"path": "large-file.bin", "stream": true}
```

Returns raw `application/octet-stream` body directly from R2 (bypasses caches).

### Stat a file

```
GET /workspace/{id}/fs/stat?path=src/index.ts
```

Response:
```json
{"size": 24, "mode": 420, "mtime": 1709900000000, "isDirectory": false}
```

### List directory

```
GET /workspace/{id}/fs/readdir?path=src
```

Response:
```json
[
  {"name": "index.ts", "isDirectory": false},
  {"name": "lib", "isDirectory": true}
]
```

Detects both explicit directories (created via `mkdir`) and implicit directories (inferred from deeper file paths).

### Create directory

```
POST /workspace/{id}/fs/mkdir
Content-Type: application/json

{"path": "src/lib", "recursive": true}
```

### Delete a file

```
POST /workspace/{id}/fs/unlink
Content-Type: application/json

{"path": "src/old.ts"}
```

### Remove directory

```
POST /workspace/{id}/fs/rmdir
Content-Type: application/json

{"path": "src/old", "recursive": true}
```

### Rename a file or directory

```
POST /workspace/{id}/fs/rename
Content-Type: application/json

{"oldPath": "src/old.ts", "newPath": "src/new.ts"}
```

Renaming a directory moves all children. R2 objects are copied to new keys and old keys deleted.

### Check existence

```
GET /workspace/{id}/fs/exists?path=src/index.ts
```

Response: `{"exists": true}`

## R2 Key Structure

```
{workspace_id}/
├── src/
│   ├── index.ts      → R2 key: "{workspace_id}/src/index.ts"
│   └── lib/
│       └── util.ts   → R2 key: "{workspace_id}/src/lib/util.ts"
├── package.json      → R2 key: "{workspace_id}/package.json"
└── .gitignore        → R2 key: "{workspace_id}/.gitignore"
```

R2 is flat key-value — the directory structure is maintained entirely in SQLite. The `files` table stores path, size, mode, mtime, and an `is_dir` flag.

## Path Validation

All paths are validated before processing:
- Max 4096 characters
- Max 256 directory depth
- No null bytes
- No traversal above root (`..` past root is rejected)
- Leading/trailing slashes stripped, double slashes collapsed
