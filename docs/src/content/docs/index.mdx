title: nodemode

**nodemode** maps Node.js primitives to Cloudflare primitives, giving you a serverless Node.js environment with real `fs`, `child_process`, and shell — things impossible on vanilla Cloudflare Workers.

## Why nodemode?

Cloudflare Workers have no `fs`, no `child_process`, and no shell. Web frameworks like Next.js work fine on Workers (via [vinext](https://github.com/cloudflare/vinext) or OpenNext), but dev tools, CLI scripts, build systems, and AI coding agents that need the filesystem and shell cannot run at all.

nodemode bridges this gap:

| Tool | What it needs | Workers alone | With nodemode |
|------|--------------|---------------|---------------|
| **zx** (Google) | Shell, pipes, exit codes | Impossible | Works |
| **simple-git** | .git/ filesystem, refs | Impossible | Works |
| **create-next-app** | mkdir, templates, npm | Impossible | Works |
| **lint-staged** | spawn linters, fs.watch | Impossible | Works |
| **AI coding agents** | fs read/write, exec | Impossible | Works |

## Mapping

| Node.js | Cloudflare Primitive | Status |
|---------|---------------------|--------|
| `fs.readFile` / `writeFile` | R2 `get()` / `put()` + SQLite cache | Working |
| `fs.readdir` / `stat` | DO SQLite directory index | Working |
| `child_process.exec` (builtins) | 24 in-DO command emulators | Working |
| `child_process.spawn` (heavy) | Cloudflare Container via `ctx.container` | Working |
| Pipes `cmd1 \| cmd2` | Built-in pipe support | Working |
| Chains `&&` `\|\|` `;` | Built-in chain support | Working |
| `stdin` / `stdout` / `stderr` | WebSocket + DO SQLite buffer | Working |
| `process.env` | Worker env / Secrets | Working |

## Quick Start

```bash
npx nodemode init
npm install
npx nodemode deploy
```

```ts
import { Workspace, createHandler } from "nodemode";

export { Workspace };
export default { fetch: createHandler() };
```

Or use the client SDK from any JavaScript environment:

```ts
import { NodeMode } from "nodemode/client";

const nm = new NodeMode("https://my-worker.workers.dev", "my-workspace");
await nm.writeFile("index.ts", "console.log('hello');");
const result = await nm.exec("cat index.ts | grep hello");
```

## How nodemode differs from vinext

**vinext** reimplements the Next.js API surface on Vite so Next.js apps can deploy to Workers. It solves "run my web framework on the edge" — web frameworks don't need `fs` or `child_process`.

**nodemode** provides `fs`, `child_process`, and shell primitives that Workers lack. It solves "run my dev tools, build systems, and CLI scripts on the edge."

They are complementary:
- vinext = run Next.js apps on Workers (no shell needed)
- nodemode = run tools that need shell + filesystem on Workers (zx, git, npm, AI agents)
