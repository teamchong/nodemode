> **Experimental** — research prototype, not production-ready.

**nodemode** is a Node.js runtime on Cloudflare Workers, built for AI coding agents (opencode, Codex, Claude Code) that need file I/O, code search, and script execution — all at $0, all on the edge, no containers required.

## The problem

Cloudflare Workers have no `fs`, no `child_process`, and no shell. Web frameworks work fine (vinext, OpenNext), but AI coding agents that need the filesystem, process spawning, and code execution cannot run at all:

| Tool | What it needs | Workers alone | With nodemode |
|------|--------------|---------------|---------------|
| **opencode** (AI agent) | fs read/write, grep -r, exec, worker_threads | Impossible | Works |
| **Claude Code** | file I/O, shell commands, JS execution | Impossible | Works |
| **zx** (Google) | Shell, pipes, exit codes | Impossible | Works |
| **simple-git** | .git/ filesystem, refs | Impossible | Works |

## Three-tier execution

```
Tier 1: Shell builtins     → execute in DO ($0, <1ms)
Tier 2: JS/TS execution    → JsRunner in DO ($0, ~ms)
Tier 3: Native binaries    → Container (last resort, ~$0.02/hr)
```

## Three-tier file cache

```
Hot:   zerobuf WASM Memory  (sub-μs, zero-copy Uint8Array views)
Warm:  SQLite file_cache    (sub-ms, persists across DO evictions)
Cold:  R2                   (~10-50ms, durable)
```

The hot tier uses [zerobuf](https://www.npmjs.com/package/zerobuf) to store file content in WebAssembly linear memory — reads return `Uint8Array` views directly into WASM memory with no copies.

## Quick Start

```bash
npx nodemode init
npm install
npx nodemode deploy
```

```ts
import { Workspace, createHandler } from "nodemode";

export { Workspace };
export default { fetch: createHandler() };
```

Or use the client SDK from any JavaScript environment:

```ts
import { NodeMode } from "nodemode/client";

const nm = new NodeMode("https://my-worker.workers.dev", "my-workspace");
await nm.writeFile("index.ts", "console.log('hello');");
const result = await nm.exec("cat index.ts | grep hello");
```

## Module Shims — The vinext Approach for Node.js APIs

Just as [vinext](https://github.com/cloudflare/vinext) shims `next/*` imports so Next.js apps run on Workers, nodemode shims `node:fs` and `node:child_process` so Node.js tools run on Workers.

Configure wrangler to alias Node.js modules to nodemode's shims:

```jsonc
// wrangler.jsonc
{
  "alias": {
    "fs": "nodemode/shims/fs",
    "child_process": "nodemode/shims/child_process"
  }
}
```

Now any library that does `import fs from "node:fs"` or `import { exec } from "node:child_process"` transparently gets nodemode's implementation — no Container needed, $0 cost, sub-μs for cached reads.

| Approach | vinext | nodemode |
|----------|--------|----------|
| Shims | `next/link`, `next/router`, `next/image` | `node:fs`, `node:child_process` |
| Enables | Next.js apps on Workers | AI coding agents on Workers |
| How | Reimplements Next.js API on Vite | Reimplements Node.js API on R2+SQLite+DO+WASM |
| Cost | $0 (Worker) | $0 (built-in commands + JsRunner in DO) |
